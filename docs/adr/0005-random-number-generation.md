# ADR 0005: Server-Authoritative Random Number Generation

- Status: Accepted
- Date: 2026-07-16
- Phase: 0

## Context

Random outcomes (gathering rewards, combat variance, restock pools) have
economic value. Client-influenced or replayable randomness enables farming
exploits; `Math.random()` is neither cryptographically strong nor auditable.

## Decision

1. **All authoritative random outcomes are generated server-side.**
   `Math.random()` is forbidden for any authoritative outcome.
2. Two approved mechanisms:
   - **Node `crypto`** (`crypto.randomInt`, `crypto.randomBytes`) for one-shot
     draws such as restock pool selection or reward rolls fixed at action
     start.
   - A **deterministic server-side PRNG** (e.g. ChaCha20- or SHA-256-based
     stream) seeded with secret server-generated bytes plus a **persisted
     counter**, for sequences that must be replayed deterministically from
     persisted state (combat resolution).
3. Seeds are never exposed to clients before the owning activity completes.
4. Outcomes decided at action start (gathering rewards) are stored
   server-private immediately and revealed only at completion, so refreshing
   or replaying cannot reroll them.

## Consequences

- Combat can be resumed and audited deterministically from its persisted
  seed and counter.
- Reward generation is testable by injecting a seeded PRNG in tests.
- A repository-wide review (and later lint rule if configured) keeps
  `Math.random()` out of authoritative code paths; it may appear only in
  cosmetic frontend effects.
